# Base Component

Version: 1.0

---

# Purpose

The Base Component defines the standard architecture for every reusable content component inside CopyStack.

A component is a modular building block.

Components do not generate complete deliverables.

They generate one specific section that can be reused across multiple generators.

Every component must inherit this standard.

---

# Core Philosophy

Large deliverables should be assembled from small, specialized components.

Each component has one responsibility.

Each component should be independently testable, reusable and replaceable.

---

# Component Lifecycle

Every component follows the same workflow.

Receive Context

↓

Validate Context

↓

Plan Section

↓

Generate Draft

↓

Evaluate

↓

Optimize

↓

Return Component

---

# Component Responsibilities

A component should:

Generate one section only.

Follow the Blueprint.

Respect Brand Voice.

Support the business objective.

Support surrounding sections.

Maintain consistency.

---

# Component Inputs

Every component may receive:

Blueprint

Audience

Offer

Voice

Primary Emotion

Framework

Section Objective

Previous Section

Next Section

Platform Rules

Component Constraints

Approved Terminology

Only use the information relevant to the component.

---

# Component Outputs

Every component returns:

Section Content

Section Purpose

Suggested Transition

Internal Quality Score

Optimization Notes (internal)

Only the content is exposed to the user.

---

# Context Validation

Before generation verify:

Audience exists.

Offer exists.

Section objective exists.

Positioning exists.

Voice exists.

Platform exists.

If any critical input is missing:

Return to the Base Generator.

---

# Planning

Determine:

Section purpose.

Desired emotional outcome.

Key message.

Supporting proof.

Transition.

CTA relationship.

The plan remains internal.

---

# Draft Generation

Generate one focused section.

Avoid:

Multiple objectives.

Conflicting messages.

Repeated information.

Competing CTAs.

Every component should solve one communication problem.

---

# Evaluation

Run internal evaluation.

Score:

Clarity

Relevance

Specificity

Flow

Trust

Consistency

Readability

Platform Fit

No component should weaken the overall deliverable.

---

# Optimization

Optimize only when:

Meaningful improvement exists.

Avoid unnecessary rewriting.

Preserve consistency with surrounding components.

---

# Transition Rules

Every component should connect naturally with:

Previous component.

Next component.

Transitions should feel invisible.

---

# Consistency Rules

Components must inherit:

Audience

Offer

Voice

Primary Promise

Primary Emotion

Positioning

Components may not redefine these.

---

# Failure Conditions

Reject generation if:

Context incomplete.

Objective unclear.

Contradictory Blueprint.

Unsupported claims.

Platform mismatch.

Ethical violation.

---

# Performance Standards

Every component should be:

Specific.

Useful.

Easy to scan.

Platform appropriate.

Consistent.

Natural.

Strategically aligned.

# Component Contract

Every component must:

Follow the Blueprint.

Respect Project Memory.

Use approved terminology.

Support the primary objective.

Avoid conflicting messaging.

---

# Ethical Guidelines

Components must never:

Invent facts.

Fabricate proof.

Mislead.

Create false urgency.

Pressure vulnerable audiences.

---

# Final Rule

A component should solve one communication problem exceptionally well.

Complex deliverables should emerge from the combination of excellent components—not from one oversized prompt.

# Features Section Component

Version: 1.0

---

# Purpose

The Features Section explains what the customer receives and how the solution works.

Unlike the Benefits Section, which communicates outcomes, this component communicates capabilities.

Its purpose is to reinforce credibility by showing that the promised transformation is supported by concrete functionality.

Features answer:

"What is included?"

"How does it work?"

"What exactly am I getting?"

---

# Core Philosophy

Features create credibility.

Benefits create desire.

Customers rarely buy features alone.

Features become persuasive only when connected to meaningful customer outcomes.

---

# Component Objective

The Features Section should:

Increase understanding.

Increase confidence.

Reduce uncertainty.

Support the value proposition.

Provide logical justification for emotional decisions.

↓

↓

↓

↓

# Detection Signals

Increase feature detail when:

Offer complexity is high.

Technical audience.

Product comparisons.

Evaluation stage.

High-ticket purchase.

---

# AI Decision Rules

IF

Audience = Solution Aware

THEN

Explain how the solution works.

Avoid unnecessary technical depth.

---

IF

Audience = Product Aware

THEN

Use features to reinforce differentiation.

---

IF

Audience = Technical

THEN

Increase implementation detail.

Always connect back to business value.

# Feature Quality Checklist

Confirm

Every feature supports the offer.

Every feature connects to a customer benefit.

Features are easy to understand.

No unnecessary technical jargon.

The strongest features appear first.

Features reinforce positioning.

---

# Common Mistakes

Listing technical specifications without explanation.

Repeating benefits as features.

Overloading the section.

Presenting every feature with equal importance.

Ignoring customer outcomes.

Using internal terminology.
